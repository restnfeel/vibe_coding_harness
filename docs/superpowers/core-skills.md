# 주요 Superpowers 스킬 상세 가이드

14개 스킬 중 핵심적인 스킬들의 동작 방식과 활용법을 상세히 알아봅니다.

## 계획 및 설계 스킬

### Brainstorming

아이디어를 **구현 전에 완전한 설계로 발전**시키는 협업 대화 스킬입니다.

#### 워크플로우

1. 기존 프로젝트 파일과 문서를 탐색하여 맥락 파악
2. 한 번에 하나씩 명확한 질문으로 요구사항 확인
3. 2~3가지 접근 방식을 트레이드오프와 함께 제안
4. 설계를 섹션별로 나누어 순차적으로 승인 획득
5. 설계 문서를 `docs/plans/YYYY-MM-DD-<topic>-design.md`에 저장
6. `writing-plans` 스킬로 구현 단계 전환

#### 핵심 원칙

- **설계 승인 전 코드 작성 금지** — "간단해 보이는" 프로젝트도 예외 없음
- 질문은 **한 번에 하나씩**, 가능하면 객관식으로 제시
- **YAGNI**(You Aren't Gonna Need It) 원칙을 적극 적용
- 대안을 항상 탐색하고, 사용자 승인 체크포인트를 통과

:::caution
Brainstorming 스킬의 핵심 게이트: 설계가 승인되기 전에는 어떤 구현 작업도 시작하지 않습니다. 프로젝트 스캐폴딩, 코드 작성, 파일 생성 모두 금지됩니다.
:::

---

### Writing Plans

승인된 설계를 **2~5분 단위의 실행 가능한 태스크**로 분해하는 스킬입니다.

#### 계획 문서 구조

```markdown
**Goal:** [한 문장으로 목표 설명]
**Architecture:** [2~3 문장으로 접근 방식 설명]
**Tech Stack:** [핵심 기술/라이브러리]

## Task 1: [태스크 이름]
- 파일: `src/lib/auth.ts`
- 실패하는 테스트 작성 → 실패 확인 → 구현 → 통과 확인 → 커밋
```

#### 작성 규칙

- 각 태스크는 **2~5분 안에 완료** 가능한 크기
- **정확한 파일 경로** 명시 (제네릭한 참조 금지)
- 모든 태스크는 **TDD 사이클**을 포함
- 실행 커맨드와 **예상 출력**까지 기술
- DRY, YAGNI, 빈번한 커밋 원칙 적용

#### 실행 경로

계획 완료 후 두 가지 선택지:
1. **Subagent-Driven** — 현재 세션에서 서브에이전트가 태스크별 실행
2. **Executing Plans** — 별도 세션에서 체크포인트 기반 실행

---

### Executing Plans

별도 세션에서 계획을 실행하며, **리뷰 체크포인트**를 거치는 스킬입니다. 대규모 작업이나 팀 환경에서 유용합니다.

## 구현 및 실행 스킬

### Test-Driven Development (TDD)

모든 구현에 **RED-GREEN-REFACTOR 사이클을 강제**하는 핵심 스킬입니다.

#### TDD 사이클

```
1. RED     — 최소한의 실패하는 테스트 작성
2. VERIFY  — 테스트가 올바른 이유로 실패하는지 확인
3. GREEN   — 테스트를 통과시키는 최소한의 코드 작성
4. VERIFY  — 테스트 통과 확인
5. REFACTOR — 테스트 녹색 상태에서 코드 정리
```

#### 철칙 (The Iron Law)

> **실패하는 테스트 없이 프로덕션 코드를 작성하지 않는다.**

테스트 전에 작성된 코드는 "참고용"으로도 남기지 않고 **전부 삭제**합니다.

#### 테스트-퍼스트 vs 테스트-애프터

| | 테스트 먼저 (TDD) | 테스트 나중 |
|---|---|---|
| 질문 | "이것이 무엇을 **해야** 하는가?" | "이것이 무엇을 **하는가**?" |
| 관점 | 요구사항 기반 | 구현 기반 |
| 편향 | 없음 | 실제 코드에 편향 |

#### 완료 체크리스트

- [ ] 모든 함수에 먼저 실패한 테스트가 있는가
- [ ] 테스트가 Mock 대신 실제 코드를 사용하는가
- [ ] 모든 테스트가 깨끗하게 통과하는가
- [ ] 엣지 케이스와 에러 상황이 커버되는가

### Subagent-Driven Development

태스크마다 **새로운 서브에이전트를 파견**하고, **2단계 리뷰**를 거치는 스킬입니다.

#### 태스크별 실행 흐름

```
┌─────────────────────────────────────────┐
│  1. 서브에이전트 파견 (구현)                │
│     └→ TDD로 태스크 구현 + 셀프 리뷰       │
│  2. Spec 준수 리뷰어 파견                  │
│     └→ 명세 대비 구현 검증                  │
│  3. 코드 품질 리뷰어 파견                  │
│     └→ 코드 품질, 패턴, 성능 검증            │
│  4. 이슈 있으면 → 1번으로 복귀              │
│  5. 양쪽 리뷰 통과 → 다음 태스크             │
└─────────────────────────────────────────┘
```

#### 핵심 규칙

- **Spec 리뷰가 코드 리뷰보다 항상 먼저** — 순서 역전 금지
- 구현 서브에이전트를 **병렬로 실행하지 않음** — 순차 실행만 허용
- main/master 브랜치에서 직접 작업 금지
- 리뷰 단계를 스킵하지 않음

---

### Dispatching Parallel Agents

**독립적인 여러 문제**를 병렬 에이전트로 동시에 처리하는 스킬입니다.

#### 사용 조건

✅ **사용할 때:**
- 3개 이상의 서로 다른 원인의 테스트 실패
- 각 문제가 독립적인 서브시스템에 존재
- 에이전트 간 상태 공유가 없음

❌ **사용하지 않을 때:**
- 실패가 연쇄적/캐스케이딩
- 전체 시스템 맥락이 필요
- 같은 파일을 수정해야 함

#### 실행 구조

1. 실패를 독립 도메인별로 그룹핑
2. 도메인당 하나의 에이전트에게 **좁은 범위 + 명확한 목표 + 제약조건** 전달
3. 병렬 파견으로 동시 조사
4. 결과 병합 후 전체 테스트 스위트 재실행

---

### Using Git Worktrees

**격리된 Git 작업 환경**을 생성하고 관리하는 스킬입니다.

#### 워크플로우

1. **디렉토리 탐지** — 기존 `.worktrees/` 확인 → CLAUDE.md 설정 확인 → 사용자에게 질문
2. **안전 검증** — `git check-ignore`로 gitignore 등록 확인, 미등록 시 자동 추가
3. **Worktree 생성** — 새 브랜치와 함께 격리된 작업 환경 생성
4. **자동 셋업** — 프로젝트 타입 감지 (package.json, Cargo.toml 등) → 의존성 설치 → 테스트 실행

:::info
Git Worktrees 스킬은 `brainstorming` (Phase 4), `subagent-driven-development`, `executing-plans` 스킬에서 자동으로 호출됩니다.
:::

## 검증 및 품질 스킬

### Systematic Debugging

**근본 원인을 먼저 파악**한 후 수정하는 4단계 디버깅 스킬입니다.

#### 4단계 프로세스

| Phase | 이름 | 핵심 활동 |
|-------|------|-----------|
| 1 | **Root Cause Investigation** | 에러 분석, 재현, 최근 변경 검토, 데이터 흐름 추적 |
| 2 | **Pattern Analysis** | 정상 동작 코드와 비교, 차이점 식별, 의존성 파악 |
| 3 | **Hypothesis & Testing** | 구체적 가설 수립, 최소 변경으로 검증, 한 번에 하나만 변경 |
| 4 | **Implementation** | 실패 테스트 작성, 단일 수정 적용, 결과 검증 |

#### 철칙

> **근본 원인 조사 없이 수정하지 않는다.**

"일단 X를 바꿔보자"는 금지됩니다. 3번 이상 수정이 실패하면, 개별 수정이 아닌 **아키텍처 자체를 재검토**합니다.

---

### Verification Before Completion

**완료를 선언하기 전 반드시 증거를 확보**하는 게이트 스킬입니다.

#### 5단계 검증 게이트

```
1. 무엇이 주장을 증명하는 커맨드인지 식별
2. 해당 커맨드를 새로 실행
3. 전체 출력과 exit code 확인
4. 출력이 주장을 뒷받침하는지 검증
5. 그제서야 완료 선언
```

#### 금지 패턴

- ❌ "아마 될 겁니다", "잘 되는 것 같습니다"
- ❌ 테스트 실행 전 만족감 표현
- ❌ 에이전트 보고를 독립 확인 없이 신뢰
- ❌ 부분 검증을 전체 검증으로 간주
- ❌ 린터 통과 = 빌드 성공으로 치환

### Code Review 스킬 (Requesting / Receiving)

#### Requesting Code Review

코드 리뷰를 **요청하고 피드백을 처리**하는 스킬입니다.

- 태스크 완료 후, 주요 기능 구현 후, 머지 전에 반드시 리뷰 요청
- 피드백 우선순위: **Critical → Important → Minor**
- Critical 이슈는 즉시 수정, Important는 진행 전 해결, Minor는 기록 후 후속 처리

#### Receiving Code Review

리뷰 피드백을 **수신하고 기술적으로 대응**하는 스킬입니다.

- 불분명하거나 기술적으로 의문이 있는 피드백은 기술적 증거로 논의
- 맹목적 수용이 아닌, 검증 후 적용

## 워크플로우 관리 스킬

### Finishing a Development Branch

작업 브랜치 완료 시 **4가지 선택지를 제시**하는 스킬입니다.

#### 절차

1. 테스트 스위트 실행 → 실패 시 진행 중단
2. 베이스 브랜치(main/master) 감지
3. 4가지 선택지 제시:

| 옵션 | 동작 |
|------|------|
| **1. 로컬 머지** | 베이스 브랜치에 머지, worktree 정리 |
| **2. PR 생성** | 리모트에 push, Pull Request 생성 |
| **3. 브랜치 보존** | 나중에 처리할 수 있도록 유지 |
| **4. 작업 폐기** | 확인 후 브랜치 삭제, worktree 정리 |

:::warning
옵션 4(작업 폐기)는 타이핑 확인을 요구합니다. 실수로 삭제하는 것을 방지합니다.
:::
