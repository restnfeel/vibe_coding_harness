Superpowers Harness — 스킬 체이닝으로 워크플로우 자동화

Superpowers의 개별 스킬들을 하나의 연속된 흐름으로 묶어, "구현해", "개발해"와 같은 단순한 키워드만으로 전체 개발 워크플로우가 자동 실행되도록 만드는 Harness 기법을 다룹니다.

Harness 기법이란?​

문제: 스킬은 있지만 흐름이 끊긴다​

Superpowers에는 14개의 스킬이 있습니다. 하지만 각 스킬을 개별적으로 호출하면 이런 상황이 발생합니다:

사용자: "로그인 기능 만들어줘"
에이전트: brainstorming 시작...
  → 설계 완료
  (멈춤) ← 다음 단계를 사용자가 지시해야 함

사용자: "이제 계획 세워줘"
에이전트: writing-plans 시작...
  → 계획 완료
  (멈춤) ← 또 다시 사용자가 지시해야 함

사용자: "이제 구현해줘"
...반복

매 단계마다 사용자가 개입해야 하니, AI 자율성의 이점이 반감됩니다.

해결: Harness로 스킬을 체이닝​

Harness 기법은 스킬들을 사전에 정의된 순서로 자동 연결하여, 하나의 키워드로 전체 워크플로우가 흐르도록 만듭니다:

사용자: "로그인 기능 개발해"
에이전트:
  1. brainstorming → 요구사항 탐색, 설계 확정
  2. using-git-worktrees → 격리 환경 생성
  3. writing-plans → 태스크 분해
  4. subagent-driven-development → TDD 기반 구현
  5. verification-before-completion → 검증
  6. finishing-a-development-branch → 머지/PR
  → 완료!

Superpowers Harness 구성​

트리거 키워드 설계​

Harness의 핵심은 트리거 키워드입니다. 사용자의 자연어 요청을 감지하여 전체 체인을 실행합니다.

트리거 유형키워드 예시실행되는 체인
신규 기능"~기능 개발해", "~만들어줘", "~구현해"Full Chain (6단계)
버그 수정"~버그 고쳐줘", "~수정해", "~디버깅해"Debug Chain (4단계)
리팩토링"~리팩토링해", "~개선해", "~정리해"Refactor Chain (5단계)

Full Chain: 신규 기능 개발​

│  └────────────────────┬────────────────────┘         │
│                       ▼                              │
│  Phase 3: 계획                                        │
│  ┌─────────────────────────────────────────┐         │
│  │ writing-plans                            │         │

│  └────────────────────┬────────────────────┘         │
│                       ▼                              │
│  Phase 4: 구현                                        │
│  ┌─────────────────────────────────────────┐         │
│  │ subagent-driven-development              │         │
│  │ ┌─────────────────────────────┐         │         │
│  │ │ 태스크별 루프:                 │         │         │
│  │ │  서브에이전트(TDD 구현)        │         │         │
│  │ │  → Spec 리뷰                  │         │         │
│  │ │  → Code 리뷰                  │         │         │
│  │ │  → 통과 시 다음 태스크          │         │         │

│  │ verification-before-completion           │         │

│  └────────────────────┬────────────────────┘         │
│                       ▼                              │
│  Phase 6: 완료                                        │
│  ┌─────────────────────────────────────────┐         │
│  │ finishing-a-development-branch           │         │

│                                                      │
└──────────────────────────────────────────────────────┘

Debug Chain: 버그 수정​

트리거: "~버그 고쳐줘" / "~수정해"

1. systematic-debugging     → 4단계 근본 원인 분석
2. test-driven-development  → 실패 테스트 작성 → 수정 → 통과
3. verification-before-completion → 전체 테스트 재확인
4. finishing-a-development-branch → 완료 처리

Refactor Chain: 코드 개선​

트리거: "~리팩토링해" / "~개선해"

1. brainstorming            → 리팩토링 범위·방향 설계
2. using-git-worktrees      → 격리 환경
3. writing-plans            → 리팩토링 태스크 분해
4. subagent-driven-development → TDD로 단계적 리팩토링
5. verification-before-completion → 기존 테스트 전체 통과 확인

Harness 적용 방법​

CLAUDE.md에 체인 규칙 정의​

프로젝트 루트의 CLAUDE.md (또는 에이전트 설정 파일)에 Harness 규칙을 선언합니다:

# Development Harness Rules

## 기능 개발 요청 시 (Full Chain)
"개발해", "만들어줘", "구현해" 키워드가 포함된 요청을 받으면:
1. brainstorming 스킬로 설계를 먼저 확정한다
2. 승인 후 git worktree를 생성한다
3. writing-plans로 2~5분 단위 태스크를 생성한다
4. subagent-driven-development로 TDD 기반 구현한다
5. verification-before-completion으로 전체 검증한다
6. finishing-a-development-branch로 완료 처리한다

## 버그 수정 요청 시 (Debug Chain)
"고쳐줘", "수정해", "디버깅" 키워드가 포함된 요청을 받으면:
1. systematic-debugging으로 근본 원인을 먼저 파악한다
2. TDD로 실패 테스트 작성 후 수정한다
3. verification-before-completion으로 검증한다
4. finishing-a-development-branch로 완료 처리한다

## 공통 규칙
- 어떤 체인이든 단계를 건너뛰지 않는다
- 각 단계의 산출물이 다음 단계의 입력이 된다
- 사용자 승인이 필요한 게이트는 반드시 대기한다

승인 게이트 (Approval Gates)​

자동 흐름이라고 해서 모든 것이 무인 실행되는 것은 아닙니다. 핵심 의사결정 지점에서는 사용자 승인을 받습니다:

게이트위치대기하는 내용
설계 승인brainstorming 완료 시설계 방향 확인
계획 승인writing-plans 완료 시태스크 목록 확인
완료 방식finishing 단계머지/PR/보존/폐기 선택

나머지 단계(worktree 생성, TDD 사이클, 리뷰, 검증)는 자동으로 진행됩니다.

일반 Superpowers vs Harness 비교​

항목일반 SuperpowersSuperpowers Harness
스킬 호출상황별 개별 활성화체인으로 연속 자동 실행
사용자 개입매 단계 지시 필요승인 게이트만 개입
워크플로우스킬 단위로 단절 가능끊김 없이 설계→구현→검증→완료
적합한 상황탐색적 작업, 부분 수정명확한 기능 개발, 체계적 작업
Harness의 핵심

Harness는 새로운 도구를 추가하는 것이 아닙니다. **이미 있는 Superpowers 스킬들을 묶는 "끈"**입니다. 개별 스킬의 품질은 그대로 유지하면서, 전체 흐름을 자동화합니다.
