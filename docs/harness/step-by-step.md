# Spec Kit Harness — 명세 체이닝으로 워크플로우 자동화

Spec Kit의 슬래시 커맨드들을 **하나의 연속된 파이프라인으로 묶어**, "설계해줘", "기획해줘"와 같은 키워드로 전체 명세 주도 개발 흐름이 자동 실행되도록 만드는 Harness 기법을 다룹니다.

## Spec Kit을 Harness로 묶는 이유

### 문제: 커맨드는 있지만 순서를 매번 기억해야 한다

Spec Kit은 8개의 슬래시 커맨드를 제공합니다. 하지만 **올바른 순서와 타이밍**을 사용자가 매번 판단해야 합니다:

```
사용자: /speckit.specify    ← 먼저 constitution을 했어야 하는데?
에이전트: (constitution 없이 명세 작성)
  → 원칙 없이 만들어진 부실한 명세

사용자: /speckit.plan       ← clarify를 안 했는데?
에이전트: (모호한 부분이 있는 채로 기술 설계)
  → 나중에 구현 단계에서 문제 발생

사용자: /speckit.implement  ← tasks를 안 만들었는데?
에이전트: (계획 없이 바로 구현)
  → 결국 바이브 코딩과 다를 바 없는 결과
```

### 해결: Harness로 커맨드 파이프라인 구성

커맨드 실행 순서를 **사전에 체인으로 정의**하면, 빠짐없이 올바른 순서로 실행됩니다:

```
사용자: "이 프로젝트 설계해줘"
에이전트:
  1. /speckit.constitution → 프로젝트 원칙 수립
  2. /speckit.specify      → 요구사항 명세
  3. /speckit.clarify      → 모호한 부분 해소
  4. /speckit.plan         → 기술 설계
  5. /speckit.tasks        → 태스크 분해
  6. /speckit.analyze      → 일관성 검증
  7. /speckit.implement    → 구현 실행
  → 완료!
```

## Spec Kit Harness 체인 구성

### 트리거 키워드 설계

| 트리거 유형 | 키워드 예시 | 실행되는 체인 |
|------------|-----------|-------------|
| **전체 개발** | "설계하고 구현해", "프로젝트 시작해" | Full Spec Chain (7단계) |
| **명세만** | "설계해줘", "기획해줘", "스펙 잡아줘" | Spec-Only Chain (4단계) |
| **구현만** | "이 스펙대로 구현해", "태스크 실행해" | Implement Chain (3단계) |

### Full Spec Chain: 설계부터 구현까지

```
┌──────────────────────────────────────────────────────┐
│  트리거: "설계하고 구현해" / "프로젝트 시작해"              │
├──────────────────────────────────────────────────────┤
│                                                      │
│  Stage 1: 원칙 수립                                    │
│  ┌─────────────────────────────────────────┐         │
│  │ /speckit.constitution                    │         │
│  │ • 아키텍처·개발·품질·운영 원칙 정의          │         │
│  │ • 산출물: constitution.md               │         │
│  └────────────────────┬────────────────────┘         │
│                       ▼                              │

│  ┌─────────────────────────────────────────┐         │
│  │ /speckit.specify                         │         │

│  │ • 산출물: spec.md                        │         │
│  └────────────────────┬────────────────────┘         │
│                       ▼                              │

│  └────────────────────┬────────────────────┘         │
│                       ▼                              │

│  ┌─────────────────────────────────────────┐         │

│  └────────────────────┬────────────────────┘         │

│  │ /speckit.analyze                         │         │
│  │ • constitution ↔ spec ↔ plan ↔ tasks    │         │
│  │ • 아티팩트 간 불일치 탐지                   │         │
│  └────────────────────┬────────────────────┘         │
│                       ▼                              │
│  Stage 7: 구현                          ──── 🔒 승인 게이트

│  │ • Foundation 완료 → User Stories 진행     │         │
│  └─────────────────────────────────────────┘         │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### Spec-Only Chain: 명세까지만

기획·설계 단계만 필요하고 구현은 나중에 하거나 다른 방법으로 할 때 사용합니다.

```
트리거: "설계해줘" / "기획해줘" / "스펙 잡아줘"

1. /speckit.constitution → 원칙 수립
2. /speckit.specify      → 요구사항 정의
3. /speckit.clarify      → 명세 보완
4. /speckit.plan         → 기술 설계
→ 산출물: constitution.md, spec.md, plan.md 생성 후 종료
```

### Implement Chain: 기존 스펙에서 구현

이미 spec.md, plan.md가 있는 상태에서 구현만 실행할 때 사용합니다.

```
트리거: "이 스펙대로 구현해" / "태스크 실행해"

1. /speckit.tasks     → 기존 plan.md 기반 태스크 분해
2. /speckit.analyze   → 일관성 검증
3. /speckit.implement → 구현 실행
```

## Harness 적용 방법

### 에이전트 설정에 체인 규칙 정의

`CLAUDE.md` 또는 프로젝트 설정 파일에 Spec Kit Harness 규칙을 선언합니다:

```markdown
# Spec Kit Harness Rules

## 전체 개발 요청 시 (Full Spec Chain)
"설계하고 구현해", "프로젝트 시작해" 키워드가 포함되면:
1. /speckit.constitution 으로 프로젝트 원칙을 먼저 수립한다
2. /speckit.specify 로 요구사항을 정의한다
3. /speckit.clarify 로 모호한 부분을 반드시 해소한다
4. /speckit.plan 으로 기술 설계를 수립하고 사용자 승인을 받는다
5. /speckit.tasks 로 태스크를 분해한다
6. /speckit.analyze 로 아티팩트 간 일관성을 검증한다
7. /speckit.implement 로 구현을 실행한다

## 기획 요청 시 (Spec-Only Chain)
"설계해줘", "기획해줘" 키워드가 포함되면:
1~4단계만 실행하고 산출물 문서를 제시한다

## 공통 규칙
- constitution이 없으면 어떤 커맨드도 실행하지 않는다
- clarify 단계를 건너뛰지 않는다 (모호함 = 구현 실패)
- plan 완료 후 반드시 Constitution Check를 수행한다
- Foundation Phase 완료 전에 User Story 태스크를 시작하지 않는다
```

### 승인 게이트

| 게이트 | 위치 | 대기하는 내용 |
|--------|------|-------------|
| **원칙 승인** | constitution 완료 시 | 프로젝트 핵심 원칙 확인 |
| **설계 승인** | plan 완료 시 | 기술 스택·아키텍처 확인 |
| **구현 시작** | implement 직전 | 태스크 목록 최종 확인 |

## 일반 Spec Kit vs Harness 비교

| 항목 | 일반 Spec Kit | Spec Kit Harness |
|------|-------------|-----------------|
| 커맨드 실행 | 개별 슬래시 커맨드 수동 호출 | 키워드 기반 자동 체이닝 |
| 순서 보장 | 사용자가 순서 기억·판단 | 체인이 올바른 순서 강제 |
| 누락 방지 | clarify, analyze 건너뛰기 쉬움 | 체인에 포함되어 자동 실행 |
| 아티팩트 연결 | 수동으로 이전 결과 참조 | 자동으로 이전 산출물을 입력으로 전달 |
| 적합한 상황 | 단일 커맨드 실행, 부분 수정 | 전체 프로젝트 개발, 체계적 작업 |

:::tip 핵심 차이
일반 Spec Kit은 **"도구 세트"**이고, Spec Kit Harness는 **"조립된 생산 라인"**입니다. 같은 도구를 사용하지만, Harness는 도구 간 연결이 사전에 설계되어 있습니다.
:::
